# Design Brief — Single-Entry MCP Skill Architecture

- **Approach:** Consolidate 13 skills into 9. `mcp` becomes the sole user-invocable skill (`user-invocable: true`, `disable-model-invocation: true`) and absorbs the three workflow skills (`mcp-new-server`, `mcp-new-client`, `mcp-audit`) as thin routing sections. `mcp-distribute` merges into `mcp-server-build` as `references/distribution.md`. The remaining 8 knowledge skills stay `user-invocable: false` (model-invocable via their descriptions).
- **Why:** The workflow skill bodies duplicated the `mcp` routing table nearly row-for-row, shared a repeated strict-rules block, and re-stated the "interview first" guard (5 occurrences). `mcp-distribute` is a lifecycle phase of server building, not a domain, and was written in a jarringly different voice. One entry point matches how the plugin is actually used: `/mcp <job>` routes; plain-English requests auto-trigger knowledge skills directly.
- **Scope:** M — rewrite 1 SKILL.md, delete 4 skill directories, add 1 reference file, touch 3 more files (server-build SKILL.md, interview description, README).
- **Constraints:** Agent Skills spec — descriptions ≤ 1024 chars and must carry trigger keywords; SKILL.md ≤ 500 lines; reference files need explicit "load when X" pointers. Knowledge skills must keep auto-triggering without `/mcp`.
- **Interface:** User: `/mcp` (menu) or `/mcp <job>`. Model: 8 knowledge skills via description matching. `mcp-server-build` description gains publish/npm/npx trigger phrases so distribution stays auto-discoverable.
- **Architecture:** `mcp` = routing table + per-intent step sequences (skill-load order + conditions only, zero SDK detail) + one Rules block. Knowledge lives in exactly one place. Deliberate near-duplicates kept because skills load independently: the one-line "interview first" guard in each build skill (serves the no-`/mcp` path) and the error-channel summary in `mcp-test` (producer vs. diagnostician audiences — confirmed distinct by reference-file scan).
- **Risks:** (1) Users with muscle memory for `/mcp-new-server` lose that command — mitigated: `/mcp new server` reads naturally. (2) `mcp-distribute`'s standalone triggers ("publish an MCP server") must not regress — mitigated via server-build description keywords. (3) Plugin cache may serve stale skills until reinstall/reload.
- **First Step:** Rewrite `skills/mcp/SKILL.md` with the merged routing + workflows.
